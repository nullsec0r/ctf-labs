Q1. Section A — VPC Flow Logs
    a) 10.0.2.44 is the payment-api-server. Who or what is 10.0.3.19,
       and why does it suddenly appear querying the payment DB at 04:04:55Z?
       Cross-reference with Evidence File 1 to answer this.
    b) The query sizes go: 248 → 18942 → 87340 → 241887 → [repeating].
       What does this escalation pattern suggest the attacker is doing?
       What is the technical term for reading data in progressively larger batches?
    c) The 6-second interval between queries is suspiciously regular.
       What does this regularity suggest about how the reads are being triggered?

Q2. Section B — Proxy Logs (this is the hardest part)
    a) Traffic to metrics.novapay.io fires every EXACTLY 120 seconds,
       payload size varies only ~8 bytes each time (1839–1847 bytes).
       What type of network behavior does this match? What is it called?
    b) At 08:01:03Z, one request suddenly sends 241,891 bytes — the same
       size as one chunk from the DB reads in Section A.
       What just happened? Connect the two evidence sources.
    c) The TLS certificate is valid and issued by Let's Encrypt, so the
       proxy didn't block it. Why is a valid TLS cert NOT proof of legitimacy?
       What should the proxy have checked instead?

Q3. Section C — DNS
    a) The real internal metrics server is metrics.novapay.internal.
       The attacker's C2 is metrics.novapay.io. Why is this naming
       choice clever from an attacker's perspective?
    b) The TTL for metrics.novapay.io is 60 seconds. The legitimate
       service has TTL=300. What does a very low TTL sometimes indicate?
       (Hint: Google "fast flux DNS")
    c) What DNS-based detection technique could have caught this?
